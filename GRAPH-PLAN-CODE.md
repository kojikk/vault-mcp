# GRAPH-PLAN-CODE.md — код-графы как неймспейсы графового слоя

> Расширение GRAPH-PLAN.md. Уточняет его §12: графы кода больше не «вне скоупа как разовый ingest отчёта», а полноценные **изолированные неймспейсы** в том же граф-слое. Цель: агент ориентируется в структуре кодовых проектов (функции/классы/вызовы) через те же graph-тулзы, но код-узлы никак не загрязняют базу знаний.

Статус: **РЕАЛИЗОВАНО и задеплоено 2026-06-13** (фазы A/B/C). 96 тестов зелёные (80 прежних + 16 новых), контейнер пересобран и рестартнут, живая проверка через SDK-клиент по HTTP пройдена: `graph_query(ns:"code:obsidian-mcp", "где собирается derived граф")` → `src/core/graph/builder.ts:58 buildDerived`. Первый код-неймспейс `code:obsidian-mcp` залит в вольт (175 узлов, 191 ребро, ts-naive экстрактор).

---

## 0. Постановка

graphify строит граф кодовой базы (узлы — функции/классы/импорты, рёбра — calls/inheritance/imports) и складывает в `graphify-out/graph.json`. Это помогает модели ориентироваться в проекте. Хотим то же самое внутри нашей системы, но:

- **информации много** — сотни-тысячи узлов на проект, это не заметки;
- **природа другая** — код-граф *derived из кода*, его не курируют руками, он пересобирается;
- **нельзя мешать** заметкам: ни в поиске, ни в индексе, ни в Obsidian graph view, ни в IDF-матчинге знаний.

Решение: **неймспейсы**. База знаний — ns `kb` (всё как сейчас, поведение по умолчанию не меняется). Каждый кодовый проект — ns `code:<project>`, отдельный файл-снапшот, отдельный IDF-корпус, отдельная сборка. Пересечение — только через явные bridge-рёбра.

## 1. Зачем это в вольте, а не в репо (честно)

Для работы *внутри* репо агенту почти так же хорошо служит локальный `graphify-out/graph.json` + их собственный MCP. Уникальная ценность хранения снапшота в вольте:

1. **Доступ без чекаута**: бот/телефон → «как у меня в wishlist-2 устроена оплата?» — ответ из код-графа без открытия репозитория.
2. **Мосты знание↔код**: решение в заметке («прямой TLS вместо CF») линкуется на конкретный модуль/функцию. `graph_path` от концепта до кода.
3. **Один интерфейс**: те же `graph_query/neighbors/path`, та же лестница чтения, тот же UNTRUSTED-рендер.

Канонический артефакт при этом может жить и в репо (`graphify-out/`, в .gitignore или нет — дело проекта); вольт хранит **синхронизированный снапшот** с зашитым commit-hash — дрейф детектируется, а не маскируется.

## 2. Хранение

```
_system/graph-edges.md                # как сейчас: kb semantic-рёбра, append-only, пишет graph_upsert (JSONL внутри .md)
_system/graph/code/<project>.jsonl    # NEW: снапшоты код-графов, пишет внешний скрипт codegraph-sync
```

> Прим. при реализации: kb-хранилище — это существующий `_system/graph-edges.md` (JSONL-строки внутри .md, чтобы не расширять write-allowlist), а не отдельный `semantic.jsonl`. Код-неймспейсы — настоящие `.jsonl`, потому что их пишет скрипт мимо MCP (сервер их только читает), так что allowlist они не затрагивают.

Свойства зоны `_system/graph/code/`:

- **Снапшот, не журнал.** Файл целиком заменяется при пересканировании (atomic tmp+rename). Никакого append/retracted — код-граф пересобираем, а не курируем. Это другой lifecycle, чем semantic.jsonl, и смешивать их нельзя.
- **Пишет не агент.** Файл кладёт скрипт синка (см. §4) напрямую на диск хоста. Правило «мутации вольта только через MCP» — про агента и контент знаний; здесь — машинный кэш, детерминированно derived из кода. vault-mcp эту зону **не пишет никогда**, только читает. Write-allowlist сервера не расширяется.
- **Git-аудит сохраняется:** скрипт синка сам делает `git add + commit` в вольте (сообщение `codegraph: <project> @ <repo-commit>`).
- **Obsidian не страдает:** `.jsonl` — не markdown, Obsidian не показывает его ни в graph view, ни в поиске. Ноль визуального шума.
- **Лимит размера** файла (как у semantic.jsonl); при превышении скрипт падает с понятной ошибкой, а не пишет обрезок.

### Схема строк (zod)

```jsonl
{"t":"meta","project":"obsidian-mcp","repo":"github.com/kojikk/obsidian-mcp","commit":"c95d4f3","scanned":"2026-06-13","generator":"graphify@1.x+adapter@1"}
{"t":"node","id":"src/core/graph/match.ts#scoreQuery","kind":"function","file":"src/core/graph/match.ts","line":42,"sig":"scoreQuery(question: string): Scored[]"}
{"t":"node","id":"src/core/graph/match.ts","kind":"module","file":"src/core/graph/match.ts"}
{"t":"edge","src":"src/core/graph/match.ts#scoreQuery","tgt":"src/core/graph/match.ts#tokenize","rel":"calls","conf":"extracted"}
```

- `id = <file>#<symbol>` — стабильный, человекочитаемый, **сразу указатель куда смотреть**: агент из ответа графа знает file:line без второго запроса.
- `kind`: `module | class | function | method | note` (note — комментарии `NOTE:/WHY:/HACK:`, которые graphify тоже извлекает — это самые ценные узлы для «зачем»).
- `rel`: `calls | imports | extends | defines | mentions`; `conf`: `extracted | inferred | ambiguous` — один в один маппится из graphify (EXTRACTED/INFERRED/AMBIGUOUS), наша confidence-модель не меняется.
- `meta.commit` + `meta.scanned` — паспорт свежести, попадает в каждый рендер.

## 3. Правки vault-mcp (ядро — неймспейсы)

| Модуль | Правка |
|---|---|
| `store.ts` | Загрузчик код-неймспейсов: скан `_system/graph/code/*.jsonl`, lazy-load при первом запросе ns, **reload по mtime** (скрипт перезаписал файл → следующий запрос видит новый граф; рестарт сервера не нужен). |
| `assemble.ts` | Сборка per-ns. kb-граф и code-графы — отдельные adjacency, отдельные пороги хабов. Память: личный масштаб (тысячи узлов на проект) — ок, но lazy-load всё равно: незапрошенный проект не грузим. |
| `match.ts` | Отдельный IDF-корпус на ns — иначе идентификаторы кода отравят матчинг знаний и наоборот. Для code-корпуса — доп. токенизация идентификаторов: split по camelCase/snake_case (`scoreQuery` → `score`, `query`), чтобы NL-вопрос «где скоринг запроса» находил функцию. |
| `traverse.ts` | Без изменений (обход не знает про ns — ему дают граф). `token_budget` уже решает проблему «информации слишком много». |
| `communities.ts` | Louvain по code-ns не гоняем в v1 (структуру кода сообщества дублируют — модули и есть сообщества). |
| `render.ts` | Рендер код-узлов: `id (kind, line N)`; шапка подграфа — паспорт свежести «снимок <scanned>, коммит <commit>». Идентификаторы кода — **такие же untrusted-строки** (имя функции может содержать что угодно): та же санитизация и обёртка. |

### Тулзы

Новых тулз — ноль либо одна. Существующие получают опциональный `ns`:

- `graph_query(question, ns?: "kb" | "code:<project>")` — дефолт `kb`, поведение не меняется. Описание дополняется: «for code structure of a known project pass ns:"code:<name>"».
- `graph_neighbors`, `graph_path` — тот же опциональный `ns`. `graph_path` работает **внутри** ns; кросс-ns путь идёт через bridge-рёбра (§5) и работает из kb-ns.
- `graph_stats` — добавляет секцию: список код-неймспейсов, узлы/рёбра, `scanned`, `commit`, возраст снапшота.

`graph_upsert` в code-ns **запрещён** (валидация: src/tgt не могут начинаться с `code:`-id… кроме bridge-кейса §5, где код-сторона — именно ссылка-строка, а не мутация code-графа).

## 4. Пайплайн извлечения — снаружи, не в сервере

tree-sitter в vault-mcp не тащим (нативные грамматики, child processes — ломает security-модель «чистый TS, no egress, один child — ripgrep»). Экстракция — на хосте, по требованию:

```
cd <репо проекта>
graphify .                                   # python, tree-sitter, локально, без LLM → graphify-out/graph.json
codegraph-sync --project wishlist-2 \
               --vault D:\Vault              # адаптер: graph.json → наш JSONL → atomic replace → git commit в вольте
```

- `codegraph-sync` — маленький скрипт (~150 строк, можно node-скриптом в этом репо: `scripts/codegraph-sync.mjs`). Читает `graphify-out/graph.json`, маппит в схему §2, добавляет meta (commit из `git rev-parse`), валидирует, пишет в вольт, коммитит.
- Запускает либо Кирилл руками, либо агент в сессии Claude Code внутри репо (Bash доступен, vault-mcp для этого не нужен). Триггер «пора пересканировать» — staleness в `graph_stats`/lint, не автоматика. Автохуки (post-commit rescan) — осознанно не в v1: скан на каждый коммит — шум.
- graphify ставится один раз (`pip install`/uv); если на проекте его нет — это не блокер для остальной системы, просто нет ns.

## 5. Мосты знание↔код (самое ценное)

Bridge-рёбра живут в **semantic.jsonl** (kb-слой, пишет агент через обычный `graph_upsert`), целевая сторона — строковый id кода:

```json
{"src":"Проекты/wishlist-2.md","tgt":"code:wishlist-2/src/bot/payments.py#create_invoice","relation":"решение реализовано в","confidence":"extracted","created":"2026-06-13","key":"..."}
```

- Из kb-стороны (`graph_query("оплата в вишлисте")`) подграф показывает bridge-ребро + **однострочный стаб** код-неймспейса: «код-граф wishlist-2: 412 узлов, снимок 2026-06-10 — детали через ns:"code:wishlist-2"». Код-узлы в kb-выдачу **не разворачиваются** — это и есть защита от затопления.
- Lint проверяет полуживые мосты: bridge-ребро указывает на id, которого нет в текущем снапшоте (функцию переименовали) → warning «мост устарел».
- Минимальная дисциплина в agent.md: при ingest решения, привязанного к коду, — одно bridge-ребро на ключевой модуль/функцию. Не на каждую функцию — только на те, про которые есть *знание*.

## 6. Гигиена изоляции (чек-лист гарантий)

Код-ns **не появляется** в:
- `search` / `read_index` / `read_hot` (digest упоминает максимум одной строкой «код-графы: N проектов» — и это под вопросом, §9);
- IDF-корпусе kb-матчинга;
- Louvain-сообществах kb;
- Obsidian (graph view, поиск, quick switcher) — гарантировано форматом `.jsonl`;
- lint-секциях про сироты/покрытие kb (у lint своя отдельная секция про code-ns: staleness + битые мосты).

## 7. Фазы

| Фаза | Содержимое | Оценка | Критерий приёмки |
|---|---|---|---|
| **A** | store/assemble/match per-ns; `ns` в graph_query/neighbors/path; секция в graph_stats; mtime-reload; тесты | 1–2 д | Рукописный фикстурный jsonl в `_system/graph/code/` → `graph_query(ns:"code:fixture")` отвечает; kb-поведение бит-в-бит прежнее (regression-тесты) |
| **B** | `scripts/codegraph-sync.mjs` + прогон на самом obsidian-mcp (dogfood) | 0.5–1 д | `graph_query("где собирается derived-граф", ns:"code:obsidian-mcp")` → указывает на builder.ts с file:line |
| **C** | Bridge-рёбра: валидация `code:`-таргетов в graph_upsert, стабы в kb-рендере, lint (staleness + битые мосты), agent.md §, описания тулз | 1 д | Вопрос о проекте из kb → ответ со стабом код-графа; битый мост виден в lint |

Деплой как всегда: build + `docker compose up -d --build` + проверка tools/list (урок 2026-06-09).

## 8. Security-чеклист

- Нет новых child processes и network-вызовов в сервере; tree-sitter/graphify — только на хосте.
- Нет новых deps в сервере (zod/graphology уже есть); скрипт синка — отдельно, в server-бандл не входит.
- `ns`-параметр валидируется: `code:[a-z0-9-]+`, маппится только внутрь `_system/graph/code/` через paths.ts — никакого traversal.
- Идентификаторы/сигнатуры/note-комментарии из кода — untrusted: санитизация + UNTRUSTED-обёртка, как у контента вольта.
- Лимит размера jsonl на файл и на суммарную загрузку в память.

## 9. Открытые вопросы

1. **Гранулярность для больших репо** (tgmaster: 3 микросервиса + фронт): один ns на репо или на сервис? v1 — на репо; если подграфы станут мусорными, `meta.subproject` + фильтр.
2. **Упоминать ли код-графы в read_hot digest** — одна строка «код-графы: 3 проекта (свежесть ok)» или вообще ничего. Мерить токены, по умолчанию — ничего.
3. **Переименования и мосты:** жёсткий id `file#symbol` ломается при рефакторинге. v1 — lint ловит битые мосты, чинит агент. Если будет болеть — fuzzy-rebind по имени символа.
4. **Свой экстрактор vs graphify:** если python-зависимость надоест, для TS-проектов можно ts-morph-экстрактор в `codegraph-sync` нативно. Не раньше, чем graphify реально начнёт мешать.

## 10. Отвергнутые альтернативы (чтобы не возвращаться)

- **Код-узлы как .md-заметки в вольте** — ровно тот страх, с которого всё началось: тысячи страниц, шум в Obsidian/поиске/индексе. Нет.
- **Монтировать репо в контейнер vault-mcp и сканировать самим сервером** — ломает security-модель (доступ за пределы VAULT_ROOT, tree-sitter-бинарь в контейнере). Нет.
- **Отдельный graphify MCP-сервер как сайдкар** — уже отвергнут в GRAPH-PLAN §0 (python-рантайм, второй сервер, нет мостов к знаниям). Для локальной работы в репо его можно поднимать ad-hoc, но это вне нашей системы.
- **Передача графа через MCP-тулзу (`graph_import` с payload)** — сотни КБ через токены агента на каждый rescan. Дорого и бессмысленно: артефакт машинный, путь «скрипт → диск» прямее.
